## 〇、EITW 核心系統架構圖分析（約 5 分鐘）

> 依據「EITW Core System Architecture / Current Application Diagram」整理

### 0.1 FAS 的核心地位

- **FAS（股務系統）** 是整個 EITW 應用程式架構的**中心**，所有股務相關的行政管理都圍繞 FAS 運作
- 圖中 FAS 與上下左右至少 **15 個系統** 有雙向箭頭連接，表示高度整合、資料交換頻繁

### 0.2 與 FAS 連接的系統（依方向）

| 方向 | 系統 | 全名/說明 | 與 FAS 的關係 |
|------|------|-----------|---------------|
| **上方** | ADM | 廠商管理平台 | 參數、通路設定 |
| | BNS | 業績獎金管理平台 | 獎金計算、通路回饋 |
| | CCC | 客服系統 | 基金主檔 IS_CCC 標記、共用資料 |
| | DMS | 投資帳務管理系統 | 投資型保單、全委、報表 |
| **左方** | EC Commerce Web | EC 電商前端 | 客戶下單介面（博揮） |
| | EC Middleware | EC 中台 | **將 EC 交易資料轉入 FAS**（博揮） |
| **下方** | SSS | 銷售管理系統 | 契約、通路、定期定額、核印 |
| | SMS | 系統管理系統 | 帳戶與權限管理（Account and permission） |
| | NAV | 基金淨值資料 | 日結、淨值計算 |
| | MDS | 行銷管理系統 | 業績、通路回饋、服務費 |
| | SG File server | 檔案伺服器 | 與 Aladdin、Fiserv、HERA 等外部交換 |
| **右方** | RightFax | 傳真系統 | 傳真扣款、文件傳送 |
| | Seal Signature Verification | 印鑑驗證系統 | 核印、憑證驗證 |
| | WEBI | 商業智慧報表 | 報表產出 |
| | Corporate Website | 公司官網 | EC API 更新基金資料 |
| | FTP→BONY | 總帳系統 | Local share folder → FTP → BONY |

### 0.3 EC 系統處理流程（博揮廠商）

```
客戶下單 → EC Commerce Web System → EC Gateway → EC API
    ↓
EC Middleware System (EC 中台)
    ↓ 「1. EC Middleware transfers EC transaction data to FAS」
FAS 股務系統（我們自建）
```

**淺顯說明**：客戶在博揮開發的 EC 網站下單，資料不會直接進 FAS，而是先經 **EC 中台** 處理與轉換，再由 EC 中台把整理好的交易資料傳到 FAS。這樣可以隔離廠商系統與核心 FAS，確保資料正確與安全。

### 0.4 從架構圖可推論的瓶頸

| 項目 | 說明 |
|------|------|
| **整合點眾多** | 任一環節出問題都可能影響 FAS |
| **EC 中台** | 中介層可能增加延遲，或成為單點故障 |
| **傳統系統** | RightFax、印鑑驗證等，可能形成技術債 |
| **多層次外部交換** | SG File server、FTP→BONY、Aladdin、Fiserv、HERA 等，資料同步與監控複雜 |
| **SMS 權限** | FAS 帳戶與權限由 SMS 管理，變更需配合 |

## 四、投信業怎麼處理 — 用「人話」講（約 10 分鐘）

### 4.1 整體流程（客戶視角）

```
客戶開戶 → 申購基金 → 定期定額/單筆買回 → 收益分配 → 對帳單
   FAS11      FAS21         FAS22/FAS27        FAS41-43      FAS62
```

### 4.2 我們 vs 博揮（EC）的分工

| 我們（FAS 自建） | 博揮（EC 廠商） |
|------------------|-----------------|
| 受益人主檔、銀行帳戶 | 網路下單介面、委託接收 |
| 申購/買回登錄、日結 | EC 交易資料寫入 ECS schema |
| 收益分配、憑證、報表 | 網路交易狀態回寫 |
| 定期定額扣款、轉申購 | 網路定期(不)定額申購 |

### 4.3 EC 整合流程（淺顯版）

> 對應架構圖：EC Commerce Web → EC Gateway/API → **EC Middleware（博揮）** → FAS

1. **客戶在博揮 EC 下單** → EC Middleware 處理後，資料寫入 `ECS.PURCHASE`、`ECS.REDEMPTION`
2. **我們用 Pipeline 轉檔** → `append_ec_purchase_to_ec_rsp_purchase.srp` 把 EC 申購轉入 `FAS.EC_RSP_PURCHASE`
3. **我們做日結** → FAS81/FAS819 的 `f_fas8192_update_ec_trades_status` 更新 EC 交易結帳狀態
4. **檢核** → `f_check_in_process_ec_tx` 檢查是否有處理中 EC 交易，避免日結衝突

### 4.4 關鍵資料表

| 資料表 | 說明 |
|--------|------|
| `FAS.HOLDER` | 受益人主檔 |
| `FAS.HOLDER_BANK_EC` | 受益人 EC 銀行帳戶 |
| `FAS.FUND` | 基金主檔（IS_EC、IS_EC_PERIODIC 等標記） |
| `ECS.PURCHASE` / `ECS.REDEMPTION` | EC 廠商交易資料 |
| `FAS.EC_RSP_PURCHASE` | 網路定期(不)定額申購，由 ECS 轉入 |

## 五、常見問題與一般狀況（約 5 分鐘）

### 5.1 依問題類型整理

| 問題類型 | 可能原因 | 處理方向 | 相關程式/資料表 |
|----------|----------|----------|------------------|
| **EC 交易狀態不同步** | 日結前仍有處理中 EC 交易 | 日結前執行 `f_check_in_process_ec_tx` 檢查 | ecs.v_authorization、fas.ec_rsp_purchase |
| **EC 申購未轉入 FAS** | Pipeline 未執行或失敗 | 檢查 append_ec_purchase_to_ec_rsp_purchase.srp | FAS25、ECS.EC_PURCHASE、FAS.EC_RSP_PURCHASE |
| **日結後 EC 狀態未更新** | 日結流程未執行 EC 更新 | 確認 f_fas8192_update_ec_trades_status 執行 | FAS81、FAS819、FAS89 |
| **定期定額扣款失敗** | 扣款磁片格式、銀行回檔 | FAS24 扣款磁片、扣款結果與銀行對帳 | FAS24 |
| **收益分配異常** | TDCC、ACH、配息平台時程 | FAS41-43 與外部系統串接時程確認 | FAS41、FAS42、FAS43 |
| **憑證作業延遲** | TDCC 集保連線 | FAS32 與 TDCC 配合 | FAS32 |
| **外部轉入失敗** | 檔案格式、時程不符 | FAS25、FAS81、FAS89 各轉入程式 | HSBC：FAS8171、FAS881-884；Dexia：FAS256-258；先鋒：FAS2536 |

### 5.2 日結前檢查清單（EC 相關）

1. 查詢 `ecs.v_authorization` 是否有 status <> 'P' 的處理中委託
2. 查詢 `fas.ec_rsp_purchase` 是否有 status <> 'P' 的處理中申購
3. 若有，需先完成該筆交易或與博揮確認後再執行日結
